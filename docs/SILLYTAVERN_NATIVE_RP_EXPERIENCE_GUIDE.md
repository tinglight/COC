# SillyTavern 原生 RP 体验指南

## 目标

在不接入本 QQ CoC bot 项目的前提下，直接体验 SillyTavern 原生功能：角色扮演流程、角色卡编辑、World Info/Lorebook 行为、提示词与上下文管理，以及日常交互手感。

本地实验地址：

```text
http://127.0.0.1:8000/
```

## 开始前

- 在 SillyTavern 里连接一个 API 服务商，例如 DeepSeek 或其它 Chat Completion 服务。
- 不要导入本项目的 CoC 模组、`.env`、SQLite 数据库或 keeper-only 资料。
- 从默认角色开始，例如 `Seraphina`，也可以新建一个干净的自定义 NPC。
- 把这次体验当成产品研究，不要当成正式备团。

## 1. 基础聊天手感

目的：了解 SillyTavern 默认 RP 聊天体验。

步骤：

1. 选择一个默认角色。
2. 发送：

   ```text
   你好，我迷路了。这里是什么地方？
   ```

3. 阅读回复。
4. 使用 swipe/regenerate 再生成一个版本。
5. 编辑上一条自己的消息，再重新生成。

观察：

- 回复是否自然？
- 是否能很快进入角色？
- swipe 是否方便挑选更好的台词？
- 编辑历史消息是否顺手？
- 界面是在帮助阅读，还是在分散注意力？

## 2. 角色卡测试

目的：了解角色卡字段如何影响说话方式。

创建一个新角色：

```text
林医生
```

建议字段：

Description：

```text
民国小城诊所医生，谨慎、观察力强，知道镇上一些传闻。
```

Personality：

```text
说话克制，常用短句，不轻易下结论。
```

Scenario：

```text
玩家深夜来到诊所，询问一名失踪病人的情况。
```

First Message：

```text
林医生把煤油灯往桌边挪了挪，抬眼看你：“这么晚来，不只是头疼吧？”
```

Example Dialogue：

```text
<START>
{{user}}: 你认识周先生吗？
{{char}}: “认识，但不熟。”林医生停了一下，“他最近来过两次，第二次没有挂号。”
```

测试消息：

```text
医生，周先生最后一次来诊所是什么时候？
```

观察：

- First Message 是否明显奠定语气？
- Example Dialogues 是否影响句子节奏？
- 自定义 NPC 是否比默认助手更稳定？
- NPC 是否会过度解释、像客服，或变得太小说化？

## 3. 世界书测试（World Info / Lorebook）

目的：了解关键词触发的上下文注入。

创建一个 World Info/Lorebook：

```text
雾桥镇公开资料
```

条目 1：

- Key：`雾桥镇`
- Content：

```text
雾桥镇是一座沿河小镇，常年多雾，旧码头和诊所是镇民最常提起的地方。
```

条目 2：

- Key：`周先生`
- Content：

```text
周先生是镇上的会计，三天前失踪。公开消息只说他最后一次出现在旧码头附近。
```

条目 3：

- Key：`旧码头`
- Content：

```text
旧码头已经废弃多年，但夜里仍有人听见船铃声。镇民通常避免在雾天靠近那里。
```

测试消息：

```text
林医生，你知道周先生和旧码头的事吗？
```

观察：

- 模型是否自然使用 lorebook？
- NPC 是否停留在公开信息范围内？
- lore 是否减少随机编造？
- 模型是否把公开事实扩展成隐藏真相？

## 4. 提示词与上下文检查

目的：感受 SillyTavern 最强也最复杂的部分。

探索：

- Main Prompt 或 Prompt Manager。
- Character Definition。
- Scenario。
- Example Messages。
- Chat History。
- World Info 插入。
- token/上下文计数器。

观察：

- 你能否看懂模型实际会收到什么？
- 哪些部分总是会被包含？
- 哪些部分是条件触发？
- 界面对上下文预算解释得清不清楚？
- 哪些控制项有用，哪些让人负担太重？

## 5. RP 工作流手感

目的：测试日常角色扮演操作。

步骤：

1. 生成一句 NPC 回复。
2. swipe/regenerate 三次。
3. 手动把其中一句 NPC 回复改成更好的版本。
4. 继续聊天，观察模型是否跟随你编辑后的版本。
5. 尝试删除或回滚一条消息。
6. 如果找到 Author's Note，添加：

   ```text
   当前场景保持克制、悬疑、桌边跑团风格。NPC 不要解释设定，只回答玩家眼前的问题。
   ```

观察：

- swipe 是否适合挑台词？
- 手动编辑是否适合作为 KP 纠偏工具？
- Author's Note 是否比改角色卡更轻便？
- SillyTavern 更适合现场跑团、备团，还是演练？

测试后，为这些问题写简短记录：

   - 简单 lore 条目？

2. 第一版应该避免什么？
   - 过多高级模型参数？

   - 复杂定时效果？
   - 过深的界面层级？

3. 最能提升本项目 NPC RP 质量的前三个功能是什么？

## 建议时间盒

```text
30 分钟：默认角色聊天 + swipe/edit
30 分钟：自定义 NPC 角色卡
30 分钟：3 条目 lorebook 测试
20 分钟：提示词/上下文界面探索
10 分钟：记录与对比
```

## 重点视角

先不要试图跑完整模组。重点观察：

- 角色卡是否让说话方式更稳定。
- 示例对话是否改善角色口吻。
- 简单 lore 条目是否减少随机编造。
- swipe/edit 是否有助于调整成桌边可用台词。
- 界面是在帮助 KP 控制 RP，还是把人埋进选项里。
